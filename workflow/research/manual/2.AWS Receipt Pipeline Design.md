# **Phase 1 Research and Discovery: AWS Ingestion and Processing Pipeline for Shopping Receipts Scanner**

## **Foundational Objectives and Architectural Intent**

The successful execution of an enterprise-grade software architecture necessitates a rigorous alignment between technical capabilities, business objectives, and external operational constraints. In the context of the digital application lifecycle, the initial Research and Discovery phase serves as the critical juncture where high-level design decisions are solidified, external dependencies are mapped, and the foundational capability development document is established.1 This phase focuses on gathering comprehensive insights and methodological research to uncover the authentic strengths, challenges, and architectural opportunities that will inform the subsequent design and deployment strategies.2 For the proposed shopping receipts scanner application, the core business objective is to automate the ingestion, optical character recognition (OCR), and intelligent data extraction of highly variable, unstructured retail receipts, transforming them into rigorously structured, machine-readable formats.

This transformation must occur with near-perfect accuracy, minimal latency, and strict cost controls. The high-level design intent dictates the implementation of a fully serverless data ingestion and processing pipeline utilizing Amazon Web Services (AWS).3 Traditional server-based architectures, which rely on persistent virtual machines or continuously running container clusters, introduce unacceptable baseline costs and operational overhead for workloads characterized by sporadic, unpredictable burst traffic—such as consumer receipt uploads that fluctuate based on time of day, shopping seasons, and user behavior.4 To achieve the required economic efficiency and operational scalability, the pipeline adopts a scale-to-zero serverless paradigm. This approach ensures that compute resources spin up instantaneously in response to distinct events, process the data, and terminate, incurring zero costs when the system is entirely idle.4

The architectural scope of this analysis is explicitly bounded by the AWS ingestion and processing backend. The pipeline begins at the secure ingestion boundary where external client applications transmit raw image or PDF artifacts, and it terminates at the persistence of the highly structured JSON payloads.6 The downstream user interface (UI) and frontend design language are governed by the Google Stitch design system—a comprehensive AI-powered UI toolkit that facilitates cohesive cross-platform experiences and generates frontend components from design parameters.7 While Google Stitch enables rapid prototyping and cohesive design logic for the web and mobile applications, the backend architecture must remain entirely agnostic to the UI implementation.9 The pipeline's sole responsibility to the frontend is to provide a secure ingestion contract and a predictable, rigidly structured data output. Similarly, downstream analytics queries, data warehouse integrations, and business intelligence dashboards fall outside the purview of this phase, requiring the pipeline to act as a pure, decoupled extraction engine.

## **The Zero-Trust Secure AWS S3 Receipts Dropbox**

The entry point for the entire ingestion pipeline is a highly secure, scalable, and rigidly controlled object storage layer. Amazon Simple Storage Service (Amazon S3) serves as the foundational data lake for all incoming receipt artifacts.3 However, exposing an S3 bucket directly to public client applications introduces severe security vulnerabilities, potential for resource exhaustion, and architectural anti-patterns. To mitigate these risks, the architecture mandates the implementation of a secure dropbox pattern, governed by temporary cryptographic credentials, strict identity and access management (IAM) bucket policies, and comprehensive ingestion constraints.

### **Authentication and Cryptographic Presigned URL Mechanics**

To facilitate secure, direct-to-storage file uploads from the mobile or web application without exposing permanent AWS credentials, the system utilizes Amazon S3 presigned URLs.10 A presigned URL is a cryptographically signed endpoint generated by a backend compute resource—typically an AWS Lambda function situated behind an Amazon API Gateway—that grants the bearer time-limited access to perform a specific action, such as an s3:PutObject request, on a specific S3 object key.11

The generation of this URL is predicated on a robust authentication flow. When a user initiates a receipt upload within the Google Stitch-designed frontend, the client application first authenticates the user via the application's centralized identity provider, such as Amazon Cognito.6 Upon successful authentication, the client requests an upload endpoint from the backend API. The authorized Lambda function, utilizing the AWS SDK, assumes an IAM execution role that possesses the requisite s3:PutObject permissions for the target ingestion bucket. The Lambda function then generates the presigned URL.11 This cryptographic signature encompasses the allowed HTTP method, the exact destination object key, the payload hash, and a strict expiration timestamp.11

Because S3 presigned URLs are inherently public resources that do not natively authenticate the bearer at the exact moment of use, the security posture of the ingestion boundary depends entirely on the robust authentication occurring prior to the URL generation and the strict limitation of the URL's validity period.13 The expiration time can and should be configured for the minimum duration necessary to complete the upload—often as little as five to fifteen minutes.11 This minimizes the window of vulnerability if the URL is intercepted. Furthermore, rather than returning the raw URL directly to the client, the backend server can be configured to respond with an HTTP 307 Temporary Redirect, seamlessly routing the client's upload request to the secure S3 endpoint while maintaining a clean API abstraction layer.14

### **Enforcing Multi-Tenant Categorization via Mandatory Object Tagging**

A critical requirement of the receipts dropbox is the mandatory association of every uploaded object with its corresponding multi-tenant identifiers, specifically user\_id and account\_id. This association is vital for downstream processing, strict data compartmentalization, privacy compliance, and granular cost allocation. Relying solely on object keys (e.g., embedding the user ID in the S3 folder path hierarchy) is insufficient for comprehensive access control and lifecycle management.15 Therefore, the system must forcefully apply S3 object tagging at the precise moment of ingestion.10

To enforce the application of the user\_id and account\_id tags, the backend Lambda function must include these tags as signed parameters when generating the presigned URL. Specifically, the x-amz-tagging HTTP header must be explicitly incorporated into the cryptographic signature.12 This introduces a specific technical nuance depending on the AWS SDK utilized. For instance, if the backend utilizes the @aws-sdk/s3-request-presigner for Node.js environments, the default behavior of the SDK is to "hoist" the x-amz-tagging header into the query string parameters of the URL.12 This hoisting process removes the tag from the X-Amz-SignedHeaders list, resulting in Amazon S3 ignoring the tags during the upload or returning a signature mismatch error.12 To rectify this, the presigner must be explicitly configured to not hoist the x-amz-tagging header.12 The header must remain a signed header, strictly requiring the client application to explicitly pass the identical x-amz-tagging header with the corresponding URL-encoded tag keys and values (e.g., user\_id=123\&account\_id=456) during the subsequent HTTP PUT request.12

To guarantee organizational compliance and completely prevent any untagged objects from persisting in the bucket, an S3 bucket policy must be implemented to enforce this tagging standard globally.16 The bucket policy utilizes a Condition block attached to the s3:PutObject action. The condition explicitly checks for the presence of the required tag keys using the s3:RequestObjectTagKeys context key, and it validates the format of the values using the s3:RequestObjectTag/\<tag-key\> condition key.17 Any client attempting to upload a receipt using a presigned URL that lacks the required x-amz-tagging header, or attempts to modify the tags to unauthorized values, will receive an HTTP 403 Access Denied error, effectively creating a zero-trust ingestion boundary.12 Furthermore, cross-account access policies can be restricted using the aws:PrincipalAccount condition key to ensure that only authorized accounts within the organizational unit can interact with the bucket.18

### **Ingestion Constraints: Rate Limits, Size Limits, and Threat Mitigation**

The ingestion pipeline must gracefully handle highly variable, localized bursts in traffic—such as thousands of users uploading holiday shopping receipts simultaneously—while vehemently protecting downstream compute resources from abnormally large payloads or malicious artifacts.

Amazon S3 inherently scales to accommodate massive request rates, supporting at least 3,500 PUT/COPY/POST/DELETE requests and 5,500 GET/HEAD requests per second per partitioned prefix.20 To maximize this throughput and avoid HTTP 503 Slow Down errors during massive ingestion events, the object key taxonomy must introduce sufficient entropy immediately following the bucket name.20 Prepending the object keys with a randomized hash or distributing them across multiple logical prefixes (rather than a sequential date-based prefix like /2026/03/24/) ensures that the internal S3 index partitions are evenly distributed, allowing the architecture to theoretically process tens of thousands of simultaneous receipt uploads without bottlenecking.15

Size constraints must be strictly enforced both during the generation of the presigned URL and via bucket policies. Receipts are typically lightweight image files or localized PDF documents. Limiting the permissible upload size to a reasonable threshold (e.g., 10 MB to 50 MB) prevents malicious actors from exhausting storage budgets or overwhelming downstream OCR services.21

Furthermore, because the application accepts unstructured files from untrusted external mobile and web clients, implementing automated malware scanning is non-negotiable.22 Treating all incoming data as untrusted traditionally required deploying complex, self-managed anti-virus pipelines with secure staging buckets.22 However, the modern architecture integrates Amazon GuardDuty Malware Protection for S3, a fully managed threat detection service that automatically scans newly uploaded objects.23 GuardDuty has expanded its scanning capabilities to support individual files up to 100 GB in size, and its archive processing capacity can handle up to 10,000 extracted files within compressed archives (up to 5 archive depth levels).23 It scans the artifacts for viruses, trojans, and malicious code before they are permitted to trigger the downstream extraction workflow.23 Alternatively, if multi-engine scanning is required for strict compliance, third-party solutions such as Antivirus for Amazon S3 by Cloud Storage Security can be deployed directly within the AWS account, utilizing engines like Sophos and ClamAV to scan files up to the maximum S3 object size of 5 TB.25 Objects flagged as malicious are immediately quarantined or permanently deleted, ensuring that the core processing pipeline remains uncompromised and operational integrity is maintained.

## **Serverless Trigger Pipeline: EventBridge vs. S3 Event Notifications**

Once a receipt is securely ingested, tagged, and verified as clean, the architecture must automatically initiate the processing workflow. The selection of the event-routing mechanism is a critical architectural decision that dictates the scalability, flexibility, and operational coupling of the entire system. The two primary candidates for this routing layer are native Amazon S3 Event Notifications and Amazon EventBridge.

### **Limitations of Legacy S3 Event Notifications**

Historically, S3 Event Notifications served as the standard building block for event-driven architectures reacting to object creations.26 However, this legacy system presents several architectural constraints that complicate large-scale, resilient application design.

First, S3 Event Notifications guarantee only "at-least-once" delivery.26 This semantic implies that the system may occasionally deliver duplicate events for a single object creation, and the events may arrive out of chronological order relative to the actual S3 operations.26 Consequently, any downstream processing logic triggered directly by these notifications must be strictly idempotent to prevent duplicate data extraction, redundant API calls, and billing anomalies.26

Second, the configuration of S3 Event Notifications is tightly coupled to the S3 bucket resource itself.15 Each event notification rule can target only a single destination—either one Amazon SNS topic, one Amazon SQS queue, or one specific AWS Lambda function.26 If the architecture requires multiple independent microservices to react to the same receipt upload, developers must construct complex fan-out architectures using intermediary SNS topics.26 Furthermore, because the notification configuration is an intrinsic property of the bucket, deploying the infrastructure via Infrastructure as Code (IaC) tools like AWS CloudFormation requires the bucket and the downstream compute targets to be managed together.15 This creates undesirable resource coupling and potential circular dependencies across different application stacks, hindering independent service deployments.15

### **The Superiority of Amazon EventBridge for Document Pipelines**

To construct a highly decoupled, advanced serverless application, the pipeline explicitly mandates the use of Amazon EventBridge as the primary event carrier for S3 object operations. EventBridge resolves the architectural limitations of legacy notifications by separating the event routing rules from the S3 bucket configuration entirely.15

By configuring the S3 bucket to send event notifications directly to the EventBridge default event bus, developers can build, scale, and deploy event-driven pipelines without constantly modifying the storage layer's configuration.21 EventBridge rules act as completely independent resources, allowing operations teams to add, remove, or modify event targets without risking disruption to the core ingestion bucket.15 This is particularly advantageous in serverless systems where independent deployment and management of resources are paramount.15

Crucially, EventBridge provides vastly superior filtering and routing capabilities compared to the rudimentary prefix and suffix matching of legacy S3 events.27 EventBridge rules can inspect the entire JSON payload of the event.21 This allows the architecture to filter events based on complex combinations of attributes, such as object size, precise time ranges, or specific event metadata fields.21 If a receipt is uploaded but lacks the required tags, or if it triggers a GuardDuty alert, EventBridge can intelligently route the event to a quarantine workflow or dead-letter queue rather than the expensive OCR pipeline.

Furthermore, a single EventBridge rule can route the same event to up to 18 different target destinations simultaneously, effortlessly enabling parallel processing workflows.21 For example, a single receipt upload event could trigger the core document processing state machine, update a centralized analytics database, and send a notification metric to a monitoring dashboard all at once.21 Finally, EventBridge offers native event archiving and replay capabilities.21 This provides a robust mechanism to reprocess historical receipt events in the case of a downstream service failure, or if a new, highly optimized AI extraction model is deployed and historical data needs to be re-evaluated.21

## **Serverless Scaling-to-Zero Compute Patterns**

The economic viability and operational sustainability of the receipt scanner rely heavily on its ability to minimize costs during periods of low or zero usage. Traditional architectures incur continuous "floor costs" just to keep compute clusters warm and responsive, resulting in significant financial waste when the system is idle at 3 AM.4 The solution adopts a strict "scale-to-zero" serverless architecture, ensuring that the system incurs absolutely no compute fees when receipts are not actively being processed.3

### **Insights from Serverless Architectural Implementations**

Analyzing reference implementations of serverless pipelines—specifically those designed for Retrieval-Augmented Generation (RAG) and document processing—reveals the mechanics of true scale-to-zero compute patterns. Implementations such as the RAGStack-Lambda framework and specific freeCodeCamp architectural patterns demonstrate how to replace always-on control planes with entirely event-driven components.3

A primary takeaway from these reference patterns is the strict avoidance of persistent virtual machines (like Amazon EC2) or continuously running container services.4 Instead, every compute step utilizes transient services that spin up exclusively when needed, run for the exact duration of the task (from seconds to a few minutes), and shut down immediately.4 For instance, asynchronous media and document processing pipelines can scale their processing times based on content type: text might process in 1 to 5 minutes, while complex OCR on high-resolution images might take 2 to 15 minutes.3 An architecture built on persistent servers would pay for the idle waiting time during these lengthy asynchronous operations. A serverless architecture sidesteps this entirely by utilizing managed orchestration and on-demand function execution.3

### **Compute Evaluation: AWS Lambda vs. AWS Fargate**

The core processing logic surrounding the AI services must be executed by dynamic compute environments. The architectural evaluation necessitates a rigorous comparison between AWS Lambda and AWS Fargate to determine the optimal compute runtime for intelligent document processing.28

| Feature | AWS Lambda | AWS Fargate (Amazon ECS) |
| :---- | :---- | :---- |
| **Execution Model** | Event-driven, function-based compute. | Container-based compute (microservices). |
| **Maximum Duration** | 15 minutes per invocation.28 | Unlimited (ideal for long-running processes).28 |
| **Scaling Mechanism** | Automatic, seamless scaling per concurrent request.5 | Container-level scaling based on defined metrics.5 |
| **Resource Allocation** | Memory-based allocation; CPU scales proportionally.5 | Independent specification of vCPU and memory.5 |
| **Idle Cost** | $0 (Scales entirely to zero).4 | High; requires active containers to avoid severe cold starts.5 |
| **Operational Overhead** | Minimal; no server management or patching.29 | Moderate; requires Docker container management and task definitions.3 |
| **Optimal Workload** | Sporadic, unpredictable, highly bursty event processing.28 | Predictable, continuous, or long-running monolithic services.5 |

An analysis of intelligent document processing workloads reveals that parsing and extracting data from standard receipts requires short, discrete bursts of compute.3 The operations primarily consist of formatting inputs, triggering external APIs, and routing outputs.

AWS Fargate, while abstracting underlying EC2 instance management and offering broad language support through containerization, still operates on a persistent container model.28 Fargate charges based on the allocated vCPU and memory for the entire lifecycle of the container, regardless of whether it is actively processing a receipt.5 For a sporadic workload like receipt uploads, maintaining warm Fargate tasks results in continuous baseline billing. Conversely, attempting to shut down Fargate tasks to zero introduces significant cold start latencies, as new containers must be provisioned, image layers pulled from registries, and the runtime initialized to handle incoming traffic.5 If a workload is small, sporadic, or does not run continuously, Fargate is vastly more expensive than Lambda.5

AWS Lambda is explicitly designed for short-lived, event-driven tasks.28 Lambda functions spin up instantaneously upon receiving an event from EventBridge, run the execution logic, and terminate automatically.4 Billing is calculated strictly per request and per millisecond of actual execution time, offering a true scale-to-zero paradigm.4 While Lambda officially supports a more limited set of native runtimes compared to Fargate (e.g., Node.js, Python, Java, Go) 29, these are more than sufficient for the required integration logic. Python 3.13+ or Node.js 24+ are industry standards for connecting to AWS AI services.3

Furthermore, Lambda provides a generous free tier of one million invocations and 400,000 GB-seconds of compute time per month.4 This makes Lambda exceptionally cost-effective for localized or highly variable workloads.29 Given that individual receipt processing coordination easily falls within the 15-minute maximum execution limit, demands immediate burstable scalability, and requires zero baseline costs, AWS Lambda is the definitive compute choice for the orchestration interactions within this pipeline.5

## **AWS Step Functions Orchestration and Pipeline Mapping**

The ingestion and processing pipeline represents a highly orchestrated, seamless flow of data from raw image artifacts residing in Amazon S3 to structured JSON intelligence. Managing the multi-step nature of this pipeline—which involves triggering complex OCR, awaiting asynchronous results, structuring the data via a Large Language Model, and persisting the final output—requires a highly reliable state machine.4 Rather than deploying a persistent orchestration engine or attempting to daisy-chain Lambda functions together (an anti-pattern that leads to high costs and fragile error handling), the system utilizes AWS Step Functions.3

Step Functions act as a serverless control plane, orchestrating the sequence of API calls and Lambda invocations using visual workflows.4 Standard workflows within Step Functions charge only per state transition ($0.025 per 1,000 transitions), meaning the orchestration layer is entirely cost-free when the system is inactive.4 This architecture replaces always-on management servers, solidifying the application's ability to maintain a zero-cost footprint during idle hours.4

### **Navigating Unpredictable Processing Times with the Callback Pattern**

The orchestration of AWS machine learning services requires handling highly unpredictable processing times. Optical Character Recognition (OCR) on high-resolution receipts, particularly those with complex tables or degraded text, can take anywhere from a few seconds to upwards of 15 minutes depending on the density of the document and the concurrent load on the AWS infrastructure.3 Implementing a synchronous polling mechanism within Step Functions—where a loop continuously checks the status of an OCR job—results in unnecessary state transitions, artificially inflates operational costs, and complicates the workflow definition.33 Furthermore, wrapping the entire process in a single synchronous Lambda function is impossible due to the 15-minute execution limit and the risk of API Gateway timeouts (which max out at 29 seconds).

To resolve this operational bottleneck, the architecture implements the Step Functions "Wait for a Callback with Task Token" integration pattern (designated by .waitForTaskToken in the Amazon States Language).34 When the state machine reaches the OCR processing step, it utilizes the AWS SDK service integration to issue a programmatic command (arn:aws:states:::aws-sdk:textract:startDocumentAnalysis.waitForTaskToken) to begin the analysis, and simultaneously pauses its own execution.35

During this suspended state, Step Functions generates a unique, cryptographic task token.33 This token is written to an Amazon DynamoDB table alongside the corresponding Textract Job ID, effectively preserving the state of the workflow without incurring continuous compute charges for idle waiting.33 The external Amazon Textract service processes the document entirely decoupled from the paused state machine.33 Once the asynchronous job completes successfully, Textract utilizes its native NotificationChannel feature to fire a completion notification directly to an Amazon SNS topic.36

This SNS message triggers a lightweight AWS Lambda callback function.33 The Lambda function reads the Job ID from the SNS message, retrieves the corresponding paused task token from the DynamoDB lookup table, fetches the raw processing results from the Textract API, and organizes the text blocks.33 It then executes the SendTaskSuccess API call back to Step Functions, passing the task token and the formatted OCR data.33 Step Functions receives the token, unpauses the specific state machine execution, and seamlessly advances to the data extraction state.35 This pattern guarantees operational consistency, prevents timeouts, and ensures the system maintains its scale-to-zero properties even during long-running asynchronous AI invocations.33 Furthermore, Step Functions natively handles input and output processing using JSONPath (or incrementally transitioning to JSONata), allowing developers to filter array data, map parameters, and pass precise JSON subsets between states without writing additional transformation code.32

## **Artificial Intelligence Extraction: AWS Textract and Amazon Nova**

The intelligence of the application resides in its ability to understand the visual semantics of a receipt and map that data to a rigid schema. This process is split into two distinct phases to optimize cost and accuracy: spatial text extraction and semantic JSON structuring.

### **Spatial and Tabular Extraction via Amazon Textract**

The first phase of the AI pipeline requires converting the visual representation of the receipt into machine-readable text. While basic text extraction can theoretically be achieved through multimodal visual embeddings (such as those offered by Amazon Nova Multimodal Embeddings), financial documents inherently rely on structured, spatial relationships, such as line items, price columns, totals, and merchant metadata.4

The pipeline utilizes Amazon Textract, a specialized machine learning service designed specifically for intelligent document processing and key information extraction (KIE).4 Textract transcends simple optical character recognition by automatically identifying and extracting complex layouts, including forms, nested tables, and implied queries.43 For the receipt scanner, the state machine invokes the asynchronous StartDocumentAnalysis API action, passing the S3 URI of the newly uploaded receipt.36 Textract natively processes scanned PDFs, high-resolution JPEGs, and PNG files, standardizing these diverse multimodal inputs into a consistent JSON representation of spatial text blocks.43

By offloading the raw character recognition and spatial layout analysis to Textract, the pipeline ensures that the downstream Large Language Model (LLM) is not burdened with pixel-level interpretation.42 Textract can accurately identify table boundaries and output them in Markdown or structured text, preserving the relationships between items, quantities, and prices.42 This dramatically reduces the input token count of the subsequent prompt and significantly lowers overall inference costs, while providing highly accurate grounding data for the LLM.42

### **Deterministic JSON Extraction via Amazon Nova and Constrained Decoding**

The ultimate objective of the ingestion pipeline is to output a rigidly structured JSON payload containing specific receipt entities (e.g., merchant name, transaction date, line items array, total amount, taxes). This JSON structure is the strict contract relied upon by the Google Stitch frontend and any downstream analytics pipelines.46 Historically, developers relied on complex prompt engineering techniques to coax foundation models into outputting valid JSON (e.g., explicitly instructing the model to output \`\`\`\`json\` and setting stop sequences).47 This heuristic approach is fundamentally flawed, fraught with reliability issues, hallucinated JSON keys, syntax errors, and the necessity of costly, recursive retry loops to correct malformed outputs.48

To achieve guaranteed, deterministic, schema-compliant output, the architecture leverages the Amazon Nova family of foundation models—specifically Amazon Nova Micro or Nova Lite, which offer blazing fast inference speeds of over 200 tokens per second.50 These models are invoked via the Amazon Bedrock Converse API.51 Crucially, the Nova models support a breakthrough architectural feature known as constrained decoding (also referred to as structured outputs).48

Constrained decoding fundamentally alters the token generation process at the foundational model level.53 Rather than allowing the LLM to select from its entire vocabulary distribution based on probability at each step, the Bedrock infrastructure dynamically compiles a grammatical rule set based on a developer-defined JSON Schema Draft 2020-12.49 When the Step Functions state machine calls the Bedrock Converse API, it specifies the required output structure using the outputConfig.textFormat parameter or by utilizing a tool definition with the strict: true flag.49

When the schema is provided, Amazon Bedrock compiles the grammar on the first run (caching successfully compiled grammars for 24 hours to eliminate latency on subsequent invocations).49 This grammar heavily constrains the model's predictive token sampling.53 For example, if the JSON schema dictates that a specific value must be a boolean, the model is physically prevented from generating any tokens other than true, false, or a valid null.48 If the schema dictates the closing of a JSON object array, the model is constrained to output solely the } or \] tokens.48

This architectural paradigm entirely eliminates the need for complex, custom parsing logic or secondary validation Lambda functions.49 It ensures 100% schema compliance, drastically reducing error rates and operational costs by eliminating failed requests and retries.49 The resulting output from the Nova model is guaranteed to be a syntactically perfect JSON object matching the exact requirements of the application, seamlessly enabling programmatic consumption by the external Google Stitch UI architecture and analytical databases.7

## **Comprehensive Compute and Cost Evaluation**

A critical mandate of serverless architectural design is ensuring that the theoretical infinite scalability does not result in unbounded financial liabilities. The chosen pipeline components offer highly granular, pay-per-use billing models. While this entirely eliminates idle costs, it requires a careful evaluation to understand the unit economics per transaction.29

### **Managing Hidden Infrastructure Costs**

While the API consumption costs of Bedrock, Textract, and Step Functions are straightforward to calculate, comprehensive cost evaluations must account for the auxiliary infrastructure required to support the serverless ecosystem.55 Real-world deployments often encounter hidden infrastructure costs that can match or even exceed the primary API billing if not carefully managed.55

The architecture minimizes these hidden costs through several strategic choices:

* **Lambda Optimization:** By utilizing AWS Step Functions direct service integrations wherever possible (e.g., calling the Bedrock Converse API directly from the state definition rather than wrapping it in a custom Lambda function), the architecture eliminates redundant compute billing.37 Lambda functions are strictly reserved for the brief, lightweight callback operations required to resume the state machine.36  
* **Decoupled State Management:** Utilizing DynamoDB for token storage during the asynchronous Textract wait state ensures that developers only pay for single read/write capacity units rather than paying for continuous compute polling, which would rapidly inflate Lambda duration costs.33  
* **Logging and Networking Constraints:** High-throughput serverless pipelines can generate massive volumes of Amazon CloudWatch logs, leading to unexpected storage and ingestion fees.55 Implementing strict log retention policies (e.g., 7 days) and reducing logging verbosity to only capture errors in production mitigates this risk. Additionally, utilizing AWS PrivateLink to route Bedrock and Textract API calls internally within the AWS network avoids public internet egress data transfer charges.56

## **Conclusion**

The Phase 1 Research and Discovery analysis confirms that a decoupled, event-driven serverless architecture heavily satisfies the complex technical requirements and business objectives of the shopping receipts scanner application. By integrating secure S3 presigned URLs with mandatory cryptographic tagging, the ingestion pipeline establishes an impenetrable security boundary that inherently categorizes user data while mitigating threats via GuardDuty. The selection of Amazon EventBridge over legacy S3 Event Notifications provides the decoupled, highly filterable routing layer necessary for resilient, extensible enterprise applications.

Furthermore, strict adherence to scale-to-zero compute patterns, utilizing AWS Lambda and AWS Step Functions, ensures that the financial model remains highly viable regardless of usage spikes or idle latency. By orchestrating a sophisticated collaboration between the spatial recognition capabilities of Amazon Textract and the deterministic, constrained decoding of Amazon Nova foundation models, the system guarantees accurate, schema-compliant JSON extraction without the burden of custom parsing logic or redundant retry loops. This pipeline establishes a highly scalable, economically optimized technological foundation ready to seamlessly deliver structured intelligence to the downstream Google Stitch user interface and enterprise analytical engines.
