# **Phase 1: Research & Discovery Master Artifact**

**Project:** AI-Native Shopping Receipts Scanner Platform

**Architecture:** AWS Serverless, Flutter, Google Stitch, Amazon Nova, AWS Textract

## **1\. Business Objectives**

The application establishes a two-sided data marketplace connecting consumer spending habits with enterprise analytics.

* **Consumer Strategy:** Implement a low-friction, gamified universal scanning model (Fetch paradigm) utilizing virtual currency to maximize Daily Active Users (DAU) and basket data capture.

* **Enterprise Strategy:** Monetize ingested data by providing Consumer Packaged Goods (CPG) brands with deterministic, zero-party, omnichannel market intelligence and anonymized peer-group benchmarking. Transform raw transactional data into actionable financial intelligence.

## **2\. High-Level Design Intent**

The architecture mandates an ultra-responsive frontend coupled with a highly scalable, zero-maintenance backend.

* **Frontend Generation Pipeline:** Utilize Google Stitch's AI-native "vibe design" to output a DESIGN.md specification. An autonomous Model Context Protocol (MCP) coding agent parses this specification to generate cross-platform Flutter code.

* **Decoupled Serverless Ingestion:** Ingest receipt images via an Amazon S3 secure dropbox utilizing cryptographically presigned URLs. Enforce mandatory user\_id and account\_id tags via IAM bucket policies and scan payloads with Amazon GuardDuty.

* **Event-Driven Orchestration:** Utilize Amazon EventBridge to decouple storage from compute, triggering an AWS Step Functions state machine to orchestrate extraction.

* **Tiered AI Extraction:** Execute initial spatial extraction using the cost-effective AWS Textract Expense API. Route complex validation to Amazon Nova Lite using strict "constrained decoding" (JSON Schema Draft 2020-12) to guarantee deterministic, database-ready JSON payloads.

* **Conversational AI:** Deploy an interactive chatbot powered by Amazon Nova 2 Lite for text queries and Nova Sonic for bidirectional WebSocket voice interactions. Implement a scale-to-zero Retrieval-Augmented Generation (RAG) pipeline via Bedrock Knowledge Bases.

## **3\. Architectural Constraints**

* **Scale-to-Zero Economics:** Always-on compute clusters, persistent virtual machines, and standing vector databases are strictly prohibited. All components must utilize pay-per-execution or pay-per-request billing.

* **UI Rendering Latency:** Mobile dashboards must render in single-digit milliseconds. Direct real-time queries to analytical OLAP backends are forbidden; data must be precomputed and served via an OLTP database.

* **Data Structure Rigidity:** AI extraction output must perfectly adhere to predefined JSON schemas to interface programmatically with downstream systems without manual validation or retry loops.

## **4\. Conflict Resolution & Architectural Decisions**

### **Conflict 1: Data Persistence and Database Selection**

* **Inconsistency:** Pipeline design specified S3 as the final persistence layer. Application design required Amazon Timestream for analytics and DynamoDB for caching.

* **Resolution:** **Hub-and-Spoke Database Architecture.** The system persists raw JSON to an Amazon S3 data lake (system of record). An EventBridge trigger fans out to three targets: an Amazon Timestream database for OLAP analytics, an Amazon DynamoDB table for sub-millisecond OLTP dashboard reads, and a vector indexing pipeline .

### **Conflict 2: Frontend-to-Backend API Boundaries**

* **Inconsistency:** Pipeline design relied on Amazon API Gateway for presigned URLs. Application design mandated AWS AppSync (GraphQL) via AWS Amplify.

* **Resolution:** **Unified GraphQL Layer via AppSync.** AWS AppSync serves as the exclusive API gateway. A dedicated GraphQL mutation resolves to an AWS Lambda function to generate the S3 Presigned URLs, maintaining a single integration paradigm for the Flutter client and global authentication via Amazon Cognito .

### **Conflict 3: RAG Vector Database Cost Constraints**

* **Inconsistency:** Standard enterprise RAG utilizes Amazon OpenSearch Serverless, which violates the scale-to-zero constraint due to a minimum 2 OCU requirement (\~$345/month idle cost).

* **Resolution:** **Aurora Serverless v2 with pgvector.** The architecture utilizes Amazon Aurora PostgreSQL Serverless v2 with the pgvector extension. This configuration scales down to 0.5 ACUs, integrates natively with Bedrock Knowledge Bases, and reduces baseline costs by up to 87% while maintaining transactional consistency.
